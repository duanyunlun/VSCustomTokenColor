# VS Code 语义 Token 配色插件（Token Styler）工作计划

## 0. 更新说明（2026-01-30）
本文档最初是“从零到 MVP”的工作计划，但当前仓库实现已明显推进到“可用 MVP（WIP）”。为避免文档误导，本次更新以代码实现为准：

1) 预览文档：已采用 `TextDocumentContentProvider` 提供 `tokenstyler-preview:` 虚拟文档，不再写入工作区文件。
2) 配置存储：当前实现以“主题 + 语言（含标准层 __standard__）”为单位存储 Preset（globalState / workspaceState），并在预览时将其合并写入 settings。
3) UI：当前顶部仅提供“保存当前配置 / 恢复当前配置”，未实现“保存/恢复所有语言预设”的独立按钮。
4) 语言列表：当前仅展示“已安装/内置”的语言（即推荐扩展已安装的项）。
5) 安全性：实现了“会话快照 + 未保存自动回滚（含下次启动兜底回滚）”。

后续计划部分仍保留，但“进度/下一步”已按实现情况重写。

## 1. 启动项目的缘由（Why now）
1) VS Code 原生设置 editor.semanticTokenColorCustomizations 为 JSON/JSONC，维护成本高、难以验证、易产生“无意覆盖/污染”。
2) 不同语言扩展（如 C#）会额外贡献 token types；用户很难系统性地发现、解释、配置，并在配置变更后快速验证效果。
3) 缺少“实时预览”的交互：用户只能反复修改 settings、打开文件对比，效率低。
4) 希望形成一套可复用的配置域：
   - 标准层（LSP 标准 token types）作为全局基础
   - 语言层（某语言扩展的 token types）覆盖基础层
   - UI 英文显示 tokenType 原名，中文显示简短语义标签（如 C#.常量标识符）

## 2. 成果定义（What we will deliver）
### 2.1 MVP（第一阶段可交付）
- 一个 VS Code 扩展，提供：
  1) **图形化编辑** foreground + bold/italic/underline（不支持 token 级 font family/size，这是 VS Code 限制）。
  2) **配置域分层**：
     - 标准 token types（LSP 23）
     - 语言 token types（以 C# 为首个适配语言）
     - 语言层覆盖标准层（同名 selector 覆盖）
  3) **实时预览**：通过“预览编辑器 + 内置代码片段”展示效果（不是取色器，而是代码片段渲染预览）。
  4) **主题绑定**：支持不同主题使用不同配置（Theme-scoped Presets）。
  5) **安全写入与回滚**：写入用户级/工作区级 settings，并提供一键恢复（restore）上一次写入前快照。

### 2.2 后续增强（非 MVP）
- 动态模式：模拟“按语言生效”（监听当前活动编辑器 languageId，动态应用对应语言层覆盖）。
- 多语言扩展适配（C/C++、TS/JS、Python…）。
- 导入/导出 Profiles（JSON）。

## 3. 范围与非范围（Scope / Non-goals）
### 3.1 Scope（做）
- 语义 token 的样式：foreground + bold/italic/underline。
- Presets：按“主题 + 语言（含标准层）”存储与编辑 token rules（当前实现不提供 Profile 管理 UI）。
- 语言 token types 列表来源：从已安装扩展 package.json contributes.semanticTokenTypes 自动发现。
- UI 本地化：根据 vscode.env.language 切换显示文本（当前实现主要体现在说明文案/提示）。

### 3.2 Non-goals（不做）
- 从 VS/VS2022 导入颜色。
- token 级别字体族/字号（VS Code 不支持）。
- 伪高亮（在 webview 内手写高亮）：预览以 VS Code 实际渲染为准。

## 4. 关键约束与风险
1) VS Code 原生 semantic token 规则**无法真正按语言隔离**（同一主题下 keyword 会影响所有语言的 keyword）。
   - 方案：MVP 先做“静态模式（按主题）”。
   - 扩展：后续提供“动态模式”模拟按语言切换（需明确副作用：会频繁写 settings，可能影响多窗口）。
2) 扩展贡献的 token types ≠ 语言服务一定输出该 token。
   - 方案：UI 提供“预览片段”+ 提示用户用 Developer: Inspect Editor Tokens and Scopes 验证。
3) 写入用户 settings 有风险。
   - 方案：写入前快照 + 一键恢复；只覆盖扩展管理的键空间。

## 5. 技术方案概览（How）
### 5.1 核心模块
- Preset 管理：按主题/语言（含标准层）存储 token rules 与字体配置；支持用户级/工作区级两套存储。
- TokenType 发现：从推荐扩展读取 `contributes.semanticTokenTypes/semanticTokenModifiers`；标准层使用 LSP 固定列表。
- Settings 应用器：
  - 静态模式：按主题写入 editor.semanticTokenColorCustomizations 的 theme block。
  - 动态模式（后续）：监听 active editor 的 languageId，切换写入规则。
- 预览系统：
  - 注册 TextDocumentContentProvider（如 tokenstyler-preview:）生成虚拟文档。
  - 预览文档展示 C#/通用代码片段；改配置后 debounce 写入 settings，预览实时变化。
  - 探测目标语言是否真的提供 semantic tokens（某些语言可能仅有 TextMate 高亮）。
- 会话回滚：
  - 编辑前快照 + 未保存自动回滚；
  - 通过 pending 标记实现下次启动兜底回滚。

### 5.2 数据结构（现行实现/建议）
- TokenStyle = { foreground?, bold?, italic?, underline? }
- PresetsByTheme = { [themeName]: { __standard__: { tokenRules }, [languageKey]: { tokenRules, fontFamily? } } }
- 说明：仓库里仍存在 Profile 相关代码，但当前主流程以 Preset 为主（Profile 暂未接入 UI）。

### 5.3 本地化策略
- 扩展侧：可用 package.nls.json + package.nls.zh-cn.json（当前实现暂未接入 contributes 文案）。
- Webview：由 extension 根据 vscode.env.language 下发/切换文案（当前实现主要用于提示/说明文案）。
- token label：
  - 计划：英文 UI 直接显示 tokenType；中文 UI 显示“语言前缀 + 简短语义”。
  - 当前实现：token 列表仍以 tokenType 为主；说明文字与部分提示会随 UI 语言切换。

## 7. 验收标准（Acceptance Criteria）
- 能在 UI 中对任意 tokenType 设置 foreground + bold/italic/underline。
- 能按主题保存多套配置，并在切换主题时生效。
- 能通过预览窗口实时看到效果。
- 不破坏用户其它 settings：
  - 仅改动扩展管理范围
  - 支持一键恢复到写入前
- 中英文显示符合要求：
  - 英文 UI：tokenType 原名
  - 中文 UI：语义标签（例如 C#.常量标识符）

## 8. 决策点（实现状态）
1) MVP 是否只做“静态模式（按主题）”，暂不做动态模式？——已采用：是。
2) 首个语言适配是否仅 C#？——当前实现：默认优先 C#，但已包含多语言列表与预览片段。
3) 写入默认目标：用户级（Global）还是工作区级（Workspace）？——当前实现：若工作区已显式配置过 `editor.semanticTokenColorCustomizations` 则默认工作区，否则默认用户；并允许在 UI 中切换。
